    52вҶ’## 1. RESUMEN EJECUTIVO
    53вҶ’
    54вҶ’PPMT (Probability Position Management Tool) es un motor de trading autГіnomo basado en **Trie progresivo de patrones (N1+N2+N3+N4)** con **fuzzy matching**, gestiГіn de capital tipo Kelly, y modo paper-trading con validaciГіn Monte Carlo antes de operar real.
    55вҶ’
    56вҶ’**Estado global (verificado 16 jun 2026):**
    57вҶ’- NГәcleo del motor (Trie, SAX, Matcher, Predictor): 100% funcional, 160 tests pasan (2 fallos conocidos no bloqueantes).
    58вҶ’- Backtesting portfolio + Monte Carlo: 100% funcional.
    59вҶ’- Trading en vivo (RealtimeTrader): 90% вҖ” paper trading funcional, persistencia de trades implementada (`storage.save_trade()`).
    60вҶ’- Dashboard FastAPI `:8420`: 100% funcional con 32 endpoints.
    61вҶ’- Pre-trade validation gate: **IMPLEMENTADO** вң… (en `/api/start-trading`, requiere verdict=PASS).
    62вҶ’- Auto-setup endpoint: **IMPLEMENTADO** вң… (`/api/auto-setup`).
    63вҶ’- Multi-token: **IMPLEMENTADO** en backtest (`/api/portfolio-backtest`, `/api/multi-setup`).
    64вҶ’- Multi-timeframe: **IMPLEMENTADO** como anГЎlisis (`/api/multi-tf-analysis`).

   791вҶ’## v0.32.4 вҖ” Fix crГӯtico: MEXC WebSocket nunca entregaba candles cerradas
   792вҶ’
   793вҶ’**Fecha:** 2026-06-17
   794вҶ’**SГӯntoma del usuario:** Tras PASS exitoso (12 trades, 41.7% WR, MC LOW RISK), al pulsar **START PAPER** el dashboard se queda parado. El terminal repite infinitamente:
   795вҶ’
   796вҶ’```
   797вҶ’Connecting to MEXC (ETH/USDT)
   798вҶ’Streaming ETH/USDT 1h
   799вҶ’Price: $1,777.07 | Position: FLAT
   800вҶ’P&L: +0.00% | Regime: ranging | Candles: 49 | Signals: 0 | Trades: 0 |
   801вҶ’Pattern: [...] | Entropy: 0.0b
   802вҶ’```
   803вҶ’
   804вҶ’`Candles: 49` **nunca** sube. `Signals: 0` **siempre**. La conexiГіn se reinicia cada 30-60s.
   805вҶ’
   806вҶ’### Root cause analysis (auditorГӯa profunda)
   807вҶ’
   808вҶ’#### Pista #1 вҖ” 49 candles = exactamente el warmup
   809вҶ’
   810вҶ’`run_live()` calcula:
   811вҶ’```python
   812вҶ’warmup_candles = cfg.sax_window_size * 2 + cfg.pattern_length * cfg.sax_window_size
   813вҶ’              = 7 * 2 + 5 * 7
   814вҶ’              = 14 + 35
   815вҶ’              = 49
   816вҶ’```
   817вҶ’
   818вҶ’Para ETH/USDT 1h: `sax_window_size=7` (de `TIMEFRAME_ALPHA_DEFAULTS["1h"]`), `pattern_length=5` (default de `LiveConfig`). El usuario ve exactamente 49 вҶ’ **0 candles vivas procesadas**. El WS estГЎ conectado (precio actualiza) pero `on_candle` nunca se invoca.
   819вҶ’
   820вҶ’#### Pista #2 вҖ” MEXC v3 kline NO tiene campo `x`
   821вҶ’
   822вҶ’`_parse_mexc_kline()` hacГӯa:
   823вҶ’```python
   824вҶ’closed=k.get("x", False)
   825вҶ’```
   826вҶ’
   827вҶ’Pero MEXC v3 kline (`spot@public.kline.v3.api`) retorna:
   828вҶ’```json
   829вҶ’{"c": "...", "d": {"e": "...", "k": {
   830вҶ’    "t": 1700000000000, "T": 1700003600000,
   831вҶ’    "s": "ETHUSDT", "i": "Min60",
   832вҶ’    "o": "1777.0", "c": "1778.5", "h": "1780.0", "l": "1776.0",
   833вҶ’    "v": "1234.5", "a": "2193000.0"
   834вҶ’}}, "s": "ETHUSDT", "t": 1700000000000}
   835вҶ’```
   836вҶ’
   837вҶ’**No hay `x`.** `closed` siempre es `False`. La condiciГіn `if candle.closed and candle.timestamp != self._last_candle_ts:` NUNCA se cumple вҶ’ `on_candle` jamГЎs se invoca вҶ’ 0 seГұales, 0 trades, 0 SAX symbols, entropy 0.0b, Pattern: `[]`.
   838вҶ’
   839вҶ’#### Pista #3 вҖ” MEXC requiere pings iniciados por el CLIENTE
   840вҶ’
   841вҶ’El cГіdigo sГіlo respondГӯa a pings del servidor:
   842вҶ’```python
   843вҶ’if msg.get("method") == "ping" or msg.get("ping"):
   844вҶ’    await ws.send(json.dumps({"method": "pong", "id": pong}))
   845вҶ’    continue
   846вҶ’```
   847вҶ’
   848вҶ’Pero MEXC v3 **no envГӯa** pings del servidor. El protocolo es:
   849вҶ’- Cliente envГӯa `{"method": "ping", "id": <id>}` cada вүӨ10s
   850вҶ’- Servidor responde `{"id": 0, "code": 0, "msg": "PONG"}` (siempre `id: 0`)
   851вҶ’
   852вҶ’Sin client pings, MEXC cierra la conexiГіn a los ~30s. El feed reconecta, pero el warmup **no** se re-ejecuta (es una sola vez en `start()`), asГӯ que el ciclo se repite para siempre вҖ” explicando los mensajes repetidos "Connecting to MEXC".
   853вҶ’
   854вҶ’#### Pista #4 вҖ” websockets library pings empeoran el problema
   855вҶ’
   856вҶ’```python
   857вҶ’async with websockets.connect(
   858вҶ’    url,
   859вҶ’    ping_interval=15,    # вҶҗ envГӯa WebSocket control-frame pings
   860вҶ’    ping_timeout=10,     # вҶҗ espera pong control-frame en 10s
   861вҶ’    ...
   862вҶ’)
   863вҶ’```
   864вҶ’
   865вҶ’`ping_interval=15` envГӯa pings a nivel de protocolo WebSocket (RFC 6455 control frames). MEXC no responde a esos (sГіlo entiende pings a nivel aplicaciГіn). Tras `ping_timeout=10` sin pong, websockets cierra la conexiГіn вҶ’ reconexiГіn вҶ’ bucle.
   866вҶ’
   867вҶ’#### Pista #5 вҖ” SUBSCRIPTION sin `id` es rechazado silenciosamente
   868вҶ’
   869вҶ’Mi test de verificaciГіn contra MEXC real (`scripts/test_mexc_ws.py`) confirmГі:
   870вҶ’```
   871вҶ’{'id': 0, 'code': 0, 'msg': 'Not Subscribed successfully! [spot@public.kline.v3.api+Min60+ethusdt]. Reason: Blocked!'}

   906вҶ’El parser por sГӯ solo no basta: la inferencia por wall-clock es frГЎgil cerca del lГӯmite de candle. AsГӯ que `_listen_mexc()` ahora:
   907вҶ’
   908вҶ’1. Mantiene `buffered_candle: Optional[Candle]` вҖ” el Гәltimo kline del periodo actual.
   909вҶ’2. Cuando llega un kline con **timestamp distinto** al del buffer, el anterior se considera cerrado:
   910вҶ’   - Se marca `prev.closed = True`
   911вҶ’   - Se invoca `on_candle(prev)` con los valores finales (ohlcv completo)
   912вҶ’   - Se reemplaza el buffer con el nuevo kline (que abre el siguiente periodo)
   913вҶ’3. Si el timestamp es el mismo, se actualiza el buffer con los Гәltimos valores (para tener el close/high/low/volume finales cuando llegue el cambio de timestamp).
   914вҶ’4. Al cerrar la conexiГіn (finally), se hace flush del Гәltimo buffer para no perder la Гәltima vela.
   915вҶ’
   916вҶ’Esto garantiza exactamente **una invocaciГіn de `on_candle` por periodo de vela**, con los valores finales.

   992вҶ’**1. Un bug silencioso puede enmascarar otro.**
   993вҶ’El usuario llevaba horas creyendo que el motor PPMT "no generaba seГұales" o "estaba muy restrictivo". En realidad, **jamГЎs llegГі ninguna vela viva al motor**. El motor veГӯa sГіlo 49 velas de warmup, no tenГӯa nada con quГ© generar seГұales. Sin telemetrГӯa de "candles vivas recibidas", fue imposible diagnosticar sin auditar el cГіdigo.
   994вҶ’
   995вҶ’**2. Asumir el formato de un exchange basГЎndote en OTRO exchange es peligroso.**
   996вҶ’El parser MEXC copiaba el campo `x` de Binance. MEXC v3 no lo tiene. AcciГіn: cada exchange debe tener tests de parser independientes que verifiquen el formato REAL de sus mensajes.
   997вҶ’
   998вҶ’**3. Las reconexiones silenciosas ocultan el problema.**
   999вҶ’El feed se reconectaba cada 30s sin log visible de "conexiГіn cerrada". El usuario sГіlo veГӯa "Connecting to MEXC" repetido, sin entender que eran reconexiones. AcciГіn: loguear explГӯcitamente cada close/reconnect con la razГіn.
  1000вҶ’
  1001вҶ’**4. Hay que distinguir protocol-level pings de application-level pings.**
  1002вҶ’WebSocket define pings a nivel de protocolo (RFC 6455 control frames). MEXC usa pings a nivel de aplicaciГіn (JSON `{"method":"ping"}`). Mezclar ambos causa desconexiones. AcciГіn: por exchange, decidir explГӯcitamente cuГЎl usar y desactivar el otro.
  1003вҶ’
  1004вҶ’### PrГіximos pasos recomendados
  1005вҶ’
  1006вҶ’1. **TelemetrГӯa de WS feed:** exponer `_candles_received`, `_ticks_received`, `_reconnects` vГӯa el dashboard para que el usuario vea si el feed estГЎ vivo.
  1007вҶ’2. **Log explГӯcito de desconexiГіn:** cuando `_listen_mexc` salga del `async for`, loguear la razГіn (ClosedOK, error, timeout).
  1008вҶ’3. **Heartbeat en dashboard:** si `_candles_received` no sube en 5 min, mostrar warning visible en el dashboard.
  1009вҶ’4. **Test de integraciГіn end-to-end:** mock MEXC WS server que envГӯe klines con timestamp cambiante, verificar que `on_candle` se invoca una vez por periodo.
  1010вҶ’5. **Soporte Binance/Bybit:** aplicar los mismos tests de parser para asegurar que no tengan bugs similares.

  1250вҶ’## v0.32.6 вҖ” State pollution entre tokens + Sweep All Tokens + UI robustez
  1251вҶ’
  1252вҶ’**Fecha:** 2026-06-17
  1253вҶ’**Commit:** prГіximo
  1254вҶ’**Bug reportado por el usuario:** "cuando pasas de uno a otro y vuelves a hacer un test si pasa o no para operar en el grafico se ven las operaciones que hizo en otro token y da como un error Step 0/5: error... VALIDATION RESULT asi debajo del boton"
  1255вҶ’
  1256вҶ’### SГӯntomas
  1257вҶ’
  1258вҶ’1. **PoluciГіn de estado entre tokens.** El usuario valida el token A (PASS o FAIL), luego selecciona el token B y vuelve a validar. En el grГЎfico del token B se ven los marcadores (entry/exit) del token A. Bajo el botГіn "Prepare Token" aparece la validaciГіn del token A ("VALIDATION RESULT" + badge PASS/FAIL del token anterior).
  1259вҶ’2. **"Step 0/5: error..." persistente.** Si una validaciГіn previa terminГі en excepciГіn, el servidor dejaba `auto_setup_status = {"step": "error", ...}` en el estado global. El frontend recibГӯa esto por WebSocket cada segundo y lo renderizaba literalmente como `Step 0/5: error...` (porque `'error'` no estГЎ en el array `['ingesting','building','backtesting','montecarlo','done']` y `indexOf` devuelve -1, entonces `currentIdx+1 = 0`).
  1260вҶ’3. **Falta de barrido masivo.** El usuario quiere un botГіn que valide todos los tokens en segundo plano para descubrir cuГЎles son operables sin tener que hacerlo uno por uno.
  1261вҶ’4. **Historial limitado.** MEXC a veces trae menos historial del necesario; el usuario pregunta si cambiar a Binance tendrГӯa mГЎs datos.
  1262вҶ’
  1263вҶ’### Causa raГӯz
  1264вҶ’
  1265вҶ’**El `terminal_state` es un singleton GLOBAL compartido por todos los tokens.** Solo hay un `validation_result`, un `auto_setup_status`, un `signals_history` y un `positions` para todo el servidor. El WebSocket broadcastea este estado cada segundo a todos los dashboards conectados. Cuando el usuario cambia de token en el dashboard, el servidor NO sabe que cambiГі вҖ” sigue enviando el estado del token anterior.
  1266вҶ’
  1267вҶ’El frontend hacГӯa:
  1268вҶ’```js
  1269вҶ’if (s.validation_result) updateValidationResult(s.validation_result);
  1270вҶ’if (s.auto_setup_status) updateSetupProgress(s.auto_setup_status);
  1271вҶ’if (s.signals_history) syncChartMarkers(s.signals_history);
  1272вҶ’```
  1273вҶ’
  1274вҶ’Sin ningГәn filtro por token. Entonces cualquier estado global se renderizaba encima del token actual sin importar a quГ© token pertenecГӯa.
  1275вҶ’
  1276вҶ’### Fix #1 вҖ” Etiquetar `auto_setup_status` y `validation_result` con `symbol`+`timeframe`
  1277вҶ’
  1278вҶ’**`server.py` `validate_token()`:**
  1279вҶ’
  1280вҶ’```python
  1281вҶ’_status_token = {"symbol": req.symbol, "timeframe": req.timeframe, "exchange": req.exchange}
  1282вҶ’
  1283вҶ’terminal_state.update_sync(
  1284вҶ’    auto_setup_status={**_status_token,
  1285вҶ’                       "step": "backtesting", "status": "running",
  1286вҶ’                       "message": f"Running backtest for {req.symbol}...",
  1287вҶ’                       "percent": 60}
  1288вҶ’)
  1289вҶ’# ... y lo mismo en cada uno de los 7 update_sync() a lo largo de la funciГіn
  1290вҶ’```
  1291вҶ’
  1292вҶ’Y al final:
  1293вҶ’```python
  1294вҶ’val_result["symbol"] = req.symbol
  1295вҶ’val_result["timeframe"] = req.timeframe
  1296вҶ’terminal_state.update_sync(validation_result=val_result, ...)
  1297вҶ’```
  1298вҶ’
  1299вҶ’Ahora cada estado lleva la firma del token que lo produjo.

  1450вҶ’### Lecciones aprendidas
  1451вҶ’
  1452вҶ’**1. Un singleton global de estado + WebSocket broadcast = poluciГіn cruzada garantizada.**
  1453вҶ’El `TerminalState` fue diseГұado para un solo trader activo. Pero el dashboard permite cambiar de token sin parar el trader anterior. Esto hace que el estado del token A "contamine" la vista del token B. **AcciГіn:** cualquier campo que pertenezca a un token especГӯfico (`validation_result`, `auto_setup_status`, `signals_history`) debe estar etiquetado con `symbol`+`timeframe` y el frontend debe filtrar por token activo antes de renderizar.
  1454вҶ’
  1455вҶ’**2. `indexOf` devuelve -1, no undefined.**
  1456вҶ’`['a','b'].indexOf('c') === -1` вҶ’ `(-1)+1 = 0` вҶ’ "Step 0/5: error..." en lugar de "ERROR". **AcciГіn:** siempre verificar `currentIdx >= 0` antes de hacer aritmГ©tica con Г©l, y manejar el caso `step === 'error'` explГӯcitamente.
  1457вҶ’
  1458вҶ’**3. Los enums de pasos deben incluir 'error' o ser tratados como error.**
  1459вҶ’El servidor tenГӯa 7 lugares que escribГӯan `step: "..."` pero solo 5 valores eran esperados por el frontend. Cualquier valor fuera del enum se renderizaba como "Step 0/5". **AcciГіn:** mantener un enum centralizado de steps vГЎlidos Y un valor `error` explГӯcito, nunca dejar que el servidor envГӯe un step no reconocido.
  1460вҶ’
  1461вҶ’**4. El "Sweep All" resuelve dos problemas a la vez.**
  1462вҶ’El usuario querГӯa descubrir quГ© tokens son operables sin tener que validar uno por uno. El sweep corre en background, salta los que ya tienen PASS (cached), y auto-registra como ChildNode los nuevos PASS вҖ” listo para operar. **AcciГіn:** cualquier feature de "descubrimiento" debe ser batcheable y correr en background con polling liviano.
  1463вҶ’
  1464вҶ’### PrГіximos pasos recomendados
  1465вҶ’
  1466вҶ’1. **Cache de validation_result por token en el frontend.** Cuando el usuario vuelve a un token ya validado, mostrar el Гәltimo resultado guardado en lugar de requerir re-validar. Necesita un endpoint `GET /api/validation/latest?symbol=X&tf=Y` que lea de la DB sin re-calcular.
  1467вҶ’2. **Auto-fallback MEXCвҶ’Binance.** Si MEXC trae <500 candles, re-intentar con Binance automГЎticamente. Mostrar un tag "data from Binance" en el chart.
  1468вҶ’3. **Persistir sweep results.** Guardar el resultado del Гәltimo sweep en DB para que el usuario pueda cerrar y reabrir el dashboard sin perder el barrido.
  1469вҶ’4. **Sweep con timeframe mГәltiple.** Permitir barrer 1h Y 4h a la vez para encontrar el mejor TF por token.

  1680вҶ’### Decisiones de diseГұo
  1681вҶ’
  1682вҶ’**1. ВҝPor quГ© `~/.ppmt/groups_config.json` en lugar de el proyecto?**
  1683вҶ’Los custom groups del usuario deben sobrevivir `git pull` y reinstalaciones. Si estuvieran en `ppmt/groups_config.json`, un `git pull` los sobreescribirГӯa. Por eso:
  1684вҶ’- `ppmt/groups_config.json` = template (se versiona, solo lectura para el usuario)
  1685вҶ’- `~/.ppmt/groups_config.json` = estado real del usuario (no se versionan)
  1686вҶ’
  1687вҶ’En el primer uso, si `~/.ppmt/groups_config.json` no existe, se copia desde el template.
  1688вҶ’
  1689вҶ’**2. ВҝPor quГ© cache 60s para tickers?**
  1690вҶ’`ccxt.fetch_tickers()` hace 1 llamada HTTP y devuelve todos los tickers del exchange (cientos). Para MEXC esto son ~600 tickers en 1 request. Cacheamos 60s para que el usuario pueda hacer click en varios grupos dinГЎmicos sin que cada click genere una nueva llamada HTTP.
  1691вҶ’
  1692вҶ’**3. ВҝPor quГ© `min_volume_usd=10M` para volatilidad/gainers/losers?**
  1693вҶ’Sin esto, los "Top 25 mГЎs volГЎtiles" estarГӯan llenos de memecoins con $1000 de volumen que hicieron +5000% en un dump. El filtro de $10M asegura que solo tokens realmente lГӯquidos aparezcan. El usuario puede subir el filtro a $50M si quiere ser mГЎs restrictivo.
  1694вҶ’
  1695вҶ’**4. ВҝPor quГ© no se implementГі "excluir correlaciГіn > 0.8"?**
  1696вҶ’Calcular la matriz de correlaciГіn entre 50 tokens requiere:
  1697вҶ’- Cargar 50 series de precios (50 llamadas API adicionales)
  1698вҶ’- Calcular 50Г—50/2 = 1225 correlaciones
  1699вҶ’- Cada click en "Load group" tardarГӯa 30+ segundos
  1700вҶ’
  1701вҶ’Para v0.33.0 se priorizГі velocidad y simplicidad. Si el usuario lo pide, se puede aГұadir en v0.34.0 con una cache diaria.
  1702вҶ’
  1703вҶ’**5. ВҝPor quГ© no se implementГі "excluir tokens con < X dГӯas de antigГјedad"?**
  1704вҶ’MEXC no expone la fecha de listing vГӯa ccxt. HabrГӯa que llamar a `/api/v3/exchangeInfo` y parsear `onboardDate` de cada sГӯmbolo вҖ” una llamada extra. Se dejГі el campo en el modelo `DEFAULT_FILTERS` (`min_listed_days`) para que el frontend lo pueda mostrar, pero el backend por ahora lo ignora (TODO v0.34.0).
  1705вҶ’
  1706вҶ’**6. ВҝPor quГ© se mantiene el fallback a 25 majors en `/api/sweep`?**
  1707вҶ’Si el usuario hace click en "Sweep" sin seleccionar grupo (o si el endpoint `groups.resolve_group` falla por API caГӯda), el sweep aГәn funciona con la lista curada. Esto es defensa en profundidad: la feature nueva nunca rompe la feature vieja.
  1708вҶ’
  1709вҶ’### PrГіximos pasos recomendados
  1710вҶ’
  1711вҶ’1. **Auto-fallback MEXC вҶ’ Binance.** Si `resolve_group()` devuelve 0 sГӯmbolos para un grupo (porque MEXC no lista todos), re-intentar con Binance y mostrar un tag "data from Binance".
  1712вҶ’2. **Cache de validation_result por token en el frontend.** Cuando el usuario vuelve a un token ya validado, mostrar el Гәltimo resultado guardado en lugar de requerir re-validar.
  1713вҶ’3. **Persistir sweep results en DB.** Tabla `sweep_results` con timestamp, group_id, symbols, verdicts вҖ” para historial.
  1714вҶ’4. **Sweep con timeframe mГәltiple.** Permitir barrer `1h + 5m + 15m` a la vez para encontrar el mejor TF por token.
  1715вҶ’5. **Endpoint `/api/validation/latest?symbol=X&tf=Y`.** Lee de la DB sin re-calcular, para que el dashboard muestre el Гәltimo verdict instantГЎneamente al cambiar de token.
  1716вҶ’6. **Soporte real para `min_listed_days`.** Implementar fetch de `onboardDate` desde `exchangeInfo` y filtrar.
  1717вҶ’7. **Modo "descubrimiento".** BotГіn "Find New Tokens" que busca tokens listados en los Гәltimos 7 dГӯas y los aГұade automГЎticamente a un grupo custom "recien_listados".

  1824вҶ’```js
  1825вҶ’const sorted = results.slice().sort((a, b) => {
  1826вҶ’  const score = (r) => r.verdict === 'PASS' ? 2 : r.verdict === 'INSUFFICIENT_DATA' ? 1 : 0;
  1827вҶ’  const tier = score(b) - score(a);
  1828вҶ’  if (tier !== 0) return tier;
  1829вҶ’  // Within the same tier: PF descending (treat undefined/NaN as 0)
  1830вҶ’  const pfA = Number.isFinite(a.profit_factor) ? a.profit_factor : 0;
  1831вҶ’  const pfB = Number.isFinite(b.profit_factor) ? b.profit_factor : 0;
  1832вҶ’  return pfB - pfA;
  1833вҶ’});

  1912вҶ’#### 1. RecalibraciГіn dinГЎmica por TF вҖ” `engine/realtime.py`
  1913вҶ’
  1914вҶ’**Antes:** `recalibration_interval: int = 2000` (fijo, sin contexto TF).
  1915вҶ’
  1916вҶ’**DespuГ©s:**
  1917вҶ’
  1918вҶ’```python
  1919вҶ’# Techo: en TFs altos el cГЎlculo da valores absurdos (526 aГұos para 1d).
  1920вҶ’# Cap a 50k velas para mantener Living Trie vivo sin recalibrar en exceso.
  1921вҶ’_RECALIBRATION_CEILING = 50_000
  1922вҶ’_RECALIBRATION_BASE = 2_000
  1923вҶ’_RECALIBRATION_REF_TF_MIN = 15  # 15m es la referencia
  1924вҶ’
  1925вҶ’def get_recalibration_interval(tf_minutes: int) -> int:
  1926вҶ’    if tf_minutes <= 0:
  1927вҶ’        return _RECALIBRATION_BASE
  1928вҶ’    factor = max(1.0, tf_minutes / _RECALIBRATION_REF_TF_MIN)
  1929вҶ’    interval = int(_RECALIBRATION_BASE * factor)
  1930вҶ’    return min(interval, _RECALIBRATION_CEILING)
  1931вҶ’```
