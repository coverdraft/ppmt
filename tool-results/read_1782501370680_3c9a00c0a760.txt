    26→## 1. Executive summary
    27→
    28→PPMT v7 is the next-generation crypto trading system that combines:
    29→
    30→- **LightGBM dual-expert regression** (LONG expert + SHORT expert, both regression on `fwd_ret_15m`)
    31→- **Sector-aware trie v2** (4 sectorial tries: blue_chip, large_cap, old_meme, new_meme — NO SAX, NO LONG/SHORT classification, NO N3/N4/N5 hierarchy)
    32→- **Online learning in 3 layers** (trie updates per candle, rolling retrain every 6h, walk-forward validation monthly)
    33→- **Adaptive SL/TP manager** (inter-trade decision: maintain, breakeven, or close based on prediction delta)
    34→- **Circuit breakers** (auto-pause on drawdown, drift, or extreme funding rate)
    35→
    36→**Goal:** LONG maintains +11% ROI/5mo (v6 baseline) while SHORT unlocks to WR > 52% (currently 45%).

   172→## 4. Trie v2 design (the new trie, no SAX)
   173→
   174→### 4.1 Why no SAX
   175→
   176→SAX (Symbolic Aggregate approXimation) discretizes continuous values to symbols 'a','b','c'. The old v0.x trie used SAX on close prices. Audit findings:
   177→
   178→- Lost information (continuous → 3-5 bins)
   179→- Required delta encoding (extra complexity)
   180→- Pattern density too high (243 max patterns at α=3, n=5 → too many sparse nodes)
   181→- Audit verdict: **SAX is obsolete** for v7
   182→
   183→### 4.2 OHLCV composite encoding (the v7 way)
   184→
   185→Per candle, compute a composite score:
   186→
   187→```python
   188→body_score = (close - open) / (high - low)         # range [-1, +1]
   189→direction = sign(close - open)                      # {-1, 0, +1}
   190→vol_signal = clip(volume / volume.rolling(20).mean(), 0.5, 5.0)  # [0.5, 5.0]
   191→
   192→composite = body_score * 0.4 + direction * 0.35 + vol_signal * 0.25
   193→```
   194→
   195→Then quantize `composite` to N bins using z-score breakpoints (training-time stats, frozen at inference):
   196→
   197→- blue_chip sector: 3 bins (low resolution, small moves)
   198→- large_cap sector: 4 bins
   199→- old_meme sector: 5 bins
   200→- new_meme sector: 6 bins (high resolution, extreme moves)
   201→
   202→**Each candle → 1 symbol** ('a','b','c',... up to N). Sequence of 5/10/15 candles → trie key.
   203→
   204→### 4.3 TrieNodeV6Metadata (8 fields, simplified)
   205→
   206→The old trie had 27 metadata fields per node. v7 uses 8:
   207→
   208→```python
   209→@dataclass
   210→class TrieNodeV6Metadata:
   211→    historical_count: int                    # total observations
   212→    sum_fwd_ret_15m: float                   # for mean calculation
   213→    sum_sq_fwd_ret_15m: float                # for variance (Welford)
   214→    last_observation_time: float             # epoch seconds
   215→    vol_regime_distribution: dict[int, int]  # {0: 12, 1: 5, 2: 8, 3: 2}
   216→    vol_regime_stats: dict[int, RegimeStatsV6]  # per-regime stats
   217→    node_type: str                           # "independent" or "dependent"
   218→    trading_observations: int                # gate: only trust if >= min_obs
   219→```
   220→
   221→**Dropped from old trie (19 fields):** trigger_candle, remaining_candles, expected_move_pct, max_drawdown_pct, max_favorable_pct, win_rate, avg_duration, sl_price, tp_price, continuation_nodes, expected_sequences, break_nodes, regime (string), regime_confidence, dominant_regime, long_stats, short_stats, move_variance, move_mean_for_variance, min_independent_count, observation_timespan, last_seen_timestamp.
   222→
   223→### 4.4 Two levels (N1 + N2), not four (N1/N2/N3/N4)
   224→
   225→**N1 — Sequence length variations (per sector, see §4.5):**
   226→- `n1_pred_3` — query trie with last 3 candles, return mean(fwd_ret_15m) of matched nodes
   227→- `n1_pred_5` — same with 5 candles
   228→
   229→**N2 — Regime-filtered:**
   230→- `n2_pred_regime_3` — same as `n1_pred_3` but only consider observations from current `vol_regime`
   231→- `n2_pred_regime_5` — same as `n1_pred_5` but only consider observations from current `vol_regime`
   232→
   233→**Why only 2 levels:** Audit found N3 (per_asset) and N4 (per_asset_regime) were structurally identical to N1/N2 — same patterns inserted in all 4 levels. The 2-level design captures the only meaningful distinction: unconditional vs regime-conditional probability.
   234→
   235→**Critical lesson from old system (v2.1 Config F):** "N3=90%, N4=0% (sparse N4 data hurts more than it helps)." → v7 must monitor N2 trie density. If N2 has < min_obs per node, fall back to N1 prediction (set `n2_pred_regime_L = n1_pred_L`).
   236→
   237→### 4.5 Sectorial tries (4 parallel)
   238→
   239→```
   240→trie_blue_chip:  BTC, ETH             — seq_len [3, 5], bins=3, min_obs=30
   241→trie_large_cap:  SOL, ADA, AVAX, LINK — seq_len [3, 5], bins=4, min_obs=20
   242→trie_old_meme:   XRP, DOGE, SHIB      — seq_len [3, 5], bins=5, min_obs=15
   243→trie_new_meme:   PEPE, WIF, BONK      — seq_len [3, 5], bins=6, min_obs=10
   244→```
   245→
   246→**Design revision (post-F5b, pre-F6):** The original master plan specified
   247→`seq_len [10, 15]` for blue_chip, `[5, 10]` for large_cap/old_meme, and `[5]`
   248→for new_meme. Mathematical audit showed this was unviable:
   249→
   250→| Sector (orig)   | Key space (bins^seq_len)  | Obs available | Obs/key (avg) | vs min_obs |
   251→|-----------------|---------------------------|---------------|---------------|------------|
   252→| blue_chip s=10  | 3^10 = 59,049             | ~234K         | ~4.0          | need 30 ❌ |
   253→| blue_chip s=15  | 3^15 = 14,348,907         | ~234K         | ~0.016        | need 30 ❌ |
   254→| large_cap s=10  | 4^10 = 1,048,576          | ~202K         | ~0.19         | need 20 ❌ |
   255→| old_meme s=10   | 5^10 = 9,765,625          | ~154K         | ~0.016        | need 15 ❌ |
   256→
   257→The original design contradicts §4.1's own critique of v0.x ("243 max patterns
   258→at α=3, n=5 → too many sparse nodes") — §4.5 proposed 59K–14M patterns, which
   259→is 240×–59,000× worse than what §4.1 deemed unacceptable.
   260→
   261→The F6 implementation (`v7_trie_conflict.py`) already worked around this by
   262→silently overriding `min_obs=30→3` (documented in its docstring: "~1% of BTC
   263→rows would have trie signal with min_obs=30 vs ~40% with min_obs=3"). This
   264→patched over the symptom but left the design broken.
   265→
   266→**Revised config** unifies all sectors to `seq_len=[3, 5]`:
   267→
   268→| Sector          | bins | s=3 key space | s=3 obs/key | s=5 key space | s=5 obs/key |
   269→|-----------------|------|---------------|-------------|---------------|-------------|
   270→| blue_chip (234K)| 3    | 27            | 8,667  ✓    | 243           | 964   ✓     |
   271→| large_cap (202K)| 4    | 64            | 3,156  ✓    | 1,024         | 197   ✓     |
   272→| old_meme (154K) | 5    | 125           | 1,232  ✓    | 3,125         | 49    ✓     |
   273→| new_meme (95K)  | 6    | 216           | 440    ✓    | 7,776         | 12    ⚠️   |
   274→
   275→All sectors now exceed `min_obs` by ≥3× at seq=3 and ≥1.2× at seq=5. The N2
   276→(regime-filtered) tier degrades gracefully via its existing N1 fallback when
   277→a specific (key, regime) bucket is sparse — no silent min_obs override needed.
   278→
   279→**F6 outcome (post-execution audit):** Trie features were extracted
   280→(`feature_observations_v7_trie`, 1.41M rows × 28 cols), integrated into
   281→materialized F6 parquets (96 features), and used to retrain LONG and SHORT
   282→experts across 5 walk-forward windows. Results vs F5 baseline:
   283→
   284→| Expert | Window   | F5 corr | F6 corr | Δ       | F5 top% | F6 top% | Δ       |
   285→|--------|----------|---------|---------|---------|---------|---------|---------|
   286→| LONG   | 2025-04  | +0.4779 | +0.4782 | +0.0003 | 52.4%   | 49.6%   | -2.8%   |

   299→**Diagnosis (why F6 was neutral):**
   300→1. `trie_conflict_3/5 = 0.0` always (std=0): N2 always falls back to N1 because
   301→   with seq_len=[3,5] every (key, regime) bucket is dense enough — the
   302→   fallback never triggers, so N1 == N2, so `|N1 - N2| = 0`.
   303→2. `trie_agreement_3 ≈ 1.0` always (std=0.018): same cause — all four
   304→   predictions (n1_pred_3/5, n2_pred_3/5) reduce to the same dense lookup.
   305→3. `trie_n1_pred_3/5` have non-trivial variance (std=0.04/0.10) but max
   306→   correlation with `fwd_ret_3` is only 0.20 (and NEGATIVE for
   307→   `trie_strength_avg`).
   308→4. Top features remain `atr_pct` (39–50%), `last_3_range_sum` (12–22%),
   309→   `range_pct` (3–4%). All three are volatility/range measures — the model
   310→   is fundamentally a volatility predictor, not a directional predictor.
   311→5. LightGBM `feature_fraction=0.85` masks ~14 features per tree; with 96
   312→   features and only ~25 carrying real signal, trie features are outcompeted
   313→   by strong v6 features.
   314→
   315→**Decision: v7 final = F5 (71 features, no trie).** The trie machinery
   316→(`v7_trie_conflict.py`, `v7_extract_trie_features.py`, sector tries, encoders)
   317→is **kept as production-ready infrastructure** for future research, but is
   318→**not loaded into the v7 shipping model**. F7 backtest will use F5a/F5b
   319→models. Guard #3 (top_feat < 30%) remains a documented limitation — the
   320→"diversify signal sources" hypothesis was falsified, and the alternative
   321→(regularize ATR via feature_fraction_bundling or interaction constraints)
   322→is deferred to a possible post-F7 iteration if backtest PnL is insufficient.
   323→
   324→**Why [3, 5] is the right length:**
   325→1. LightGBM already captures longer horizons via `ret_5`, `ret_10`, `ret_15`,
   326→   `log_ret_5/10` features — trie must complement, not duplicate.
   327→2. seq=3 captures "current candle + 2 previous" — sufficient microstructure
   328→   for momentum/reversal detection at 5m TF.
   329→3. seq=5 captures 25 minutes of context — enough for short-term regime shifts
   330→   without exploding the key space.
   331→4. Sector differentiation now comes from `bins` (3/4/5/6) and `min_obs`
   332→   (30/20/15/10) — both retained from the original design.
   333→
   334→**Why sectorial, not per-token:**
   335→- 12 tokens × 2 seq_len × 4 regimes = 96 sub-tries → still sparse for new_meme
   336→- 4 sectors × 2 seq_len × 4 regimes = 32 sub-tries → manageable, dense
   337→- Tokens within a sector share microstructure (PEPE/WIF/BONK pump/dump similarly)
   338→- LightGBM sees `sector_one_hot` feature and learns fine-grained per-token adjustments
   339→
   340→### 4.6 Asset classification mapping
   341→
   342→| Sector | Tokens | Rationale |

   353→## 5. LightGBM dual experts
   354→
   355→### 5.1 Why dual experts
   356→
   357→A single LightGBM struggles with SHORT because:
   358→- 60% of training labels are positive (bull regime 2024-2025)
   359→- Model biases toward LONG predictions
   360→- SHORT signals get diluted
   361→
   362→**Solution:** train two separate LightGBM models:
   363→
   364→| Expert | Training filter | Output | Threshold |
   365→|--------|-----------------|--------|-----------|
   366→| LONG expert | `fwd_ret_15m > 0` only | `pred_long` (always positive) | `thr_long = 0.30%` (tunable) |
   367→| SHORT expert | `fwd_ret_15m < 0` only | `pred_short` (always negative) | `thr_short = 0.40%` (tunable, more selective) |
   368→
   369→### 5.2 Sample weighting (carried from v6)
   370→
   371→Both experts use:
   372→- `2x weight` for observations where `fwd_ret_15m` is a drop (large negative)
   373→- `2x weight` for observations in BEAR_2022 window
   374→- Compounds to `4x` for bear-drops (the rare-but-critical cases)
   375→
   376→### 5.3 Hyperparameters (frozen from v6)
   377→
   378→```python
   379→params = {
   380→    "objective": "regression",
   381→    "metric": ["rmse", "l1"],
   382→    "num_leaves": 31,
   383→    "learning_rate": 0.05,
   384→    "n_estimators": 300,
   385→    "early_stopping": 30,
   386→    "min_child_samples": 50,
   387→    "reg_alpha": 0.1,
   388→    "reg_lambda": 1.0,
   389→    "random_state": 42,
   390→    "verbose": -1,
   391→}
   392→```
   393→
   394→### 5.4 Anti-leakage guards (mandatory, never disable)
   395→
   396→After every train, verify:
   397→- `top_feat_gain < 30%` of total gain (no single feature dominates)
   398→- `train_corr < 0.85` (model not overfit)
   399→- Walk-forward: train on months 1-N, test on month N+1, never the reverse
   400→
   401→If any guard fails: **reject the model, keep the previous one, alert**.
   402→
   403→### 5.5 Decision layer (final signal)
   404→
   405→```python
   406→def decide(pred_long, pred_short, trie_agreement, sector, funding_rate_z, thr_long, thr_short):
   407→    # LONG signal
   408→    if pred_long > thr_long and trie_agreement > 0.6:
   409→        return "LONG"
   410→    
   411→    # SHORT signal (more restrictive)
   412→    if abs(pred_short) > thr_short and trie_agreement > 0.7 and funding_rate_z > 1.5:
   413→        return "SHORT"
   414→    
   415→    return "WAIT"
   416→```

   420→## 6. Online learning — 3 layers
   421→
   422→### 6.1 Layer 1 — Trie online (every 5m)
   423→
   424→**Trigger:** every 5m candle close + 15m delay (to know `fwd_ret_15m` outcome)
   425→
   426→**Process:**
   427→```
   428→T = 10:00:00  Vela cierra
   429→T = 10:00:01  Consultar trie para features de ESTA vela (sin leakage)
   430→T = 10:00:02  LightGBM predice pred_long, pred_short
   431→T = 10:00:03  Decisión: LONG / SHORT / WAIT
   432→T = 10:00:04  Si trade: abrir posición
   433→T = 10:15:00  Vela + 15m cierra → sabemos outcome real
   434→T = 10:15:01  ★ INSERTAR en trie: (vela_10:00, sector, régimen) → fwd_ret_real
   435→T = 10:15:02  Posición cerrada (exit +15m)
   436→T = 10:15:03  Registrar PnL
   437→```
   438→
   439→**CRITICAL RULE — INSERT-AFTER-PREDICT:**
   440→The trie insertion happens 15 minutes AFTER the prediction. Never before. This is enforced by code structure (separate function calls, never inline).
   441→
   442→**Trie hygiene:**
   443→- `prune()` every 1000 insertions — remove nodes with <3 observations
   444→- Time decay half-life = 24h (stale patterns lose weight exponentially)
   445→- Max nodes per sector: 100K (LRU eviction if exceeded)
   446→
   447→### 6.2 Layer 2 — Rolling retrain (every 6h)
   448→
   449→**Trigger:** every 6h (00:00, 06:00, 12:00, 18:00 UTC)
   450→
   451→**Process:**
   452→1. Load last 30 days of data (12 tokens × 30 days × 288 candles = 103K candles)
   453→2. Calculate features v7 (72 features per candle)
   454→3. Walk-forward split:
   455→   - Train: days 1-25 (75K candles)
   456→   - Validation: days 26-28 (10K candles) — early stopping
   457→   - Test: days 29-30 (8K candles) — final acceptance gate
   458→4. Train LightGBM-LONG expert (filter: fwd_ret > 0)
   459→5. Train LightGBM-SHORT expert (filter: fwd_ret < 0)
   460→6. Validate on test set:
   461→   - LONG WR > 60%? → accept
   462→   - SHORT WR > 50%? → accept
   463→   - If either fails → reject, keep previous model, log alert
   464→7. Hot-swap atomic: replace `.txt` model files with file lock
   465→
   466→**Anti-overfitting protections:**
   467→- 30 days context = 103K obs. One trade moves model < 0.001%.
   468→- Early stopping on validation (never test).
   469→- If model rejected 3 times in a row → trigger Layer 3 early.
   470→
   471→### 6.3 Layer 3 — Walk-forward monthly
   472→
   473→**Trigger:** 1st of every month
   474→
   475→**Process:**
   476→1. Load last 90 days of data (309K candles)
   477→2. Run 5 walk-forward windows (monthly)
   478→3. Full anti-leakage guards verification
   479→4. Generate report: per-window WR, PF, PnL, Sharpe, drift metrics
   480→5. Decision: continue with current config / adjust hyperparams / alert human

   485→
   486→### 7.1 Inter-trade decision logic
   487→
   488→While a position is open, every 5m:
   489→
   490→```python
   491→def adaptive_check(position, current_features):
   492→    # Recalculate predictions with updated features
   493→    pred_long_new = lightgbm_long.predict(current_features)
   494→    pred_short_new = lightgbm_short.predict(current_features)
   495→    
   496→    if position.direction == "LONG":
   497→        # Reinforces: keep, extend TP
   498→        if pred_long_new > pred_long_entry * 0.8:
   499→            position.extend_tp()
   500→        
   501→        # Weakens but doesn't contradict: move SL to breakeven
   502→        elif pred_long_new > 0 and pred_long_new < pred_long_entry * 0.5:
   503→            position.move_sl_to_breakeven()
   504→        
   505→        # Contradicts: close early
   506→        elif pred_short_new < -0.15:  # predicts drop > 0.15%
   507→            position.close("early_exit_contradiction")
   508→    
   509→    elif position.direction == "SHORT":
   510→        # Symmetric logic
   511→        ...
   512→```

   515→
   516→- Don't close early if estimated loss < 0.05% (not worth the transaction cost)
   517→- Only close early if model predicts opposite direction > 0.15%
   518→- Max 1 early close per position (avoid churn)

   655→## 11. Anti-leakage rules (NON-NEGOTIABLE)
   656→
   657→These rules are baked into the code. Violating them = critical bug.
   658→
   659→### 11.1 Temporal leakage
   660→
   661→- **INSERT-AFTER-PREDICT:** Trie insertion at T+15m, prediction at T. Never the reverse.
   662→- **Train/test split:** Always by timestamp. `train.ts < test.ts_min`. No random shuffling.
   663→- **Feature computation:** `fwd_ret_*` always uses `shift(-N)`. Never `shift(N)`.
   664→- **Rolling stats:** `rolling(N).mean()` uses `closed='left'` (excludes current bar).
   665→
   666→### 11.2 Model overfitting
   667→
   668→- **Top feature guard:** `top_feat_gain < 30%` of total. If violated, regularize more.
   669→- **Train correlation guard:** `corr(pred_train, y_train) < 0.85`. If violated, reduce `num_leaves`.
   670→- **Walk-forward:** Train on months 1-N, test on month N+1. Never peek ahead.
   671→- **Early stopping:** On validation set, never test.
   672→
   673→### 11.3 Online learning safety
   674→
   675→- **No per-trade retrain:** Always 30+ days of context. One trade cannot move the model.
   676→- **Drift monitor:** If `|pred_avg - outcome_avg| > 0.5%` (24h rolling), force retrain.
   677→- **Circuit breaker:** 3 losses in a row → pause. Don't try to "win it back".
   678→
   679→### 11.4 SHORT-specific safety
   680→
   681→- **Funding rate gate:** SHORT only if `funding_rate_z > 1.5` (longs overleveraged).
   682→- **Higher threshold:** `thr_short = 0.40%` (vs `thr_long = 0.30%`). More selective.
   683→- **Trie agreement:** SHORT requires `trie_agreement > 0.7` (vs 0.6 for LONG).
   684→- **Max SHORT exposure:** 30% of concurrent positions can be SHORT.

  1001→**Online loop (master plan §6.1) implemented:**
  1002→
  1003→```
  1004→For each candle (sorted by ts):
  1005→  1. If there's a pending prediction from 15m ago → commit_outcome
  1006→  2. Lookup_pattern(features) → trie feedback (mean_outcome, n_obs)
  1007→  3. LightGBM predicts pred_long
  1008→  4. Ensemble: final_pred = 0.8*lgb + 0.2*trie_mean (if n_obs >= 5)
  1009→  5. Apply decision rule (LONG / SHORT / WAIT)
  1010→  6. predict_and_record(features, ts, symbol) → buffer for commit in 15m
  1011→  7. Compute PnL
  1012→```
  1013→
  1014→**Trie hygiene:**
  1015→- LRU eviction when nodes > max_nodes (default 100K, sim uses 2M)
  1016→- Explicit prune() to remove low-obs nodes (caller-controlled)
  1017→- Time decay half-life = 24h (effective_obs decays as 0.5^(elapsed_hours / half_life))
  1018→- Vectorized quantization with NumPy broadcasting (5 bins × 71 features → SHA1 hash key)

  1039→### 16.14 Honest assessment of F8
  1040→
  1041→**The trie online does NOT improve metrics in this configuration.** Online metrics are slightly worse than static (Sharpe 2.14 vs 2.17, PnL +246.55% vs +249.83%), and MaxDD worsened dramatically (-26.70% vs -7.63%).
  1042→
  1043→**Why F8 didn't help:**
  1044→
  1045→1. **Trie hits are rare (60-115 per window out of 1500+ trades):** Even with n_bins=2 (forcing more collisions), most candles produce a unique feature hash. The trie only kicks in for ~5% of trades.
  1046→
  1047→2. **Few observations per node:** With 100K nodes and 100K inserts, average is 1 obs/node. The `trie_min_obs=5` filter excludes most patterns, so the ensemble rarely activates.
  1048→
  1049→3. **Noise in low-obs nodes:** When the trie DOES activate (n_obs ≥ 5), the mean_outcome is computed from very few samples — it adds noise rather than signal. This is why MaxDD worsened: a few bad ensemble nudges pushed some winning trades into losing territory.
  1050→
  1051→4. **LightGBM is already strong:** The v7.5 model captures most of the predictable signal. There's little residual for the trie to add.
  1052→
  1053→**What WOULD make F8 valuable (future work):**
  1054→
  1055→- **Use fewer features for the trie key** — e.g., top 8-10 important features (rsi_14, atr_pct, ret_3, vol_std_10, btc_ret_5m) → ~3^8 = 6561 buckets, much higher collision rate
  1056→- **PCA-then-bin** — reduce 71 features to 5-8 principal components, then bin → denser trie
  1057→- **Per-sector tries** — separate trie per sector (blue_chip, large_cap, etc.) so patterns don't cross-contaminate
  1058→- **Larger min_obs threshold** — only activate ensemble when n_obs ≥ 20+ (statistical significance)
  1059→- **Use trie as a FILTER, not ensemble** — block trades where trie_mean disagrees strongly with LGB direction (asymmetric risk control)
  1060→
  1061→**Conclusion:** F8 is **implemented, tested, and reproducible**, but the simple ensemble approach does not improve on the static baseline. The component is ready to plug into F9-F13, but should be re-tuned (or replaced with a filter approach) before going live.
