    29→
    30→### Razones
    31→1. El Next.js `package.json` dice `name: cryptoquant-terminal v0.3.0` — es código heredado de otro proyecto, no es PPMT real.

    52→## 1. RESUMEN EJECUTIVO
    53→
    54→PPMT (Probability Position Management Tool) es un motor de trading autónomo basado en **Trie progresivo de patrones (N1+N2+N3+N4)** con **fuzzy matching**, gestión de capital tipo Kelly, y modo paper-trading con validación Monte Carlo antes de operar real.
    55→
    56→**Estado global (verificado 16 jun 2026):**
    57→- Núcleo del motor (Trie, SAX, Matcher, Predictor): 100% funcional, 160 tests pasan (2 fallos conocidos no bloqueantes).
    58→- Backtesting portfolio + Monte Carlo: 100% funcional.
    59→- Trading en vivo (RealtimeTrader): 90% — paper trading funcional, persistencia de trades implementada (`storage.save_trade()`).
    60→- Dashboard FastAPI `:8420`: 100% funcional con 32 endpoints.
    61→- Pre-trade validation gate: **IMPLEMENTADO** ✅ (en `/api/start-trading`, requiere verdict=PASS).
    62→- Auto-setup endpoint: **IMPLEMENTADO** ✅ (`/api/auto-setup`).
    63→- Multi-token: **IMPLEMENTADO** en backtest (`/api/portfolio-backtest`, `/api/multi-setup`).
    64→- Multi-timeframe: **IMPLEMENTADO** como análisis (`/api/multi-tf-analysis`).
    65→- Next.js `:3000`: **OBSOLETO** — ver sección de decisión arriba.

   145→## 4. COSAS QUE FALTAN ⚠️ (LISTA PRIORIZADA)
   146→
   147→### 🔴 PRIORIDAD CRÍTICA — Seguridad de Trading
   148→
   149→#### F1. Tabla `trades` en SQLite + persistencia
   150→**Estado:** NO IMPLEMENTADO
   151→**Archivos afectados:** `src/ppmt/data/storage.py`, `src/ppmt/risk/money_manager.py`
   152→
   153→El `MoneyManager` mantiene las posiciones en memoria. Cuando se reinicia el dashboard o el proceso, **se pierden todas las posiciones abiertas y el historial completo de trades**. No hay tabla `trades` en SQLite.
   154→
   155→**Qué hacer:**
   156→- Crear tabla `trades` con campos: `id, symbol, side, entry_time, entry_price, exit_time, exit_price, qty, pnl, pnl_pct, fee, status, signal_meta, asset_class`
   157→- `MoneyManager.open_position()` debe escribir un row con `status='open'`
   158→- `MoneyManager.close_position()` debe actualizar el row con `status='closed'` y los datos de salida
   159→- Método `Storage.get_trade_history(symbol=None, limit=100)` para consultar historial
   160→
   161→#### F2. Pre-trade Validation Gate (Backtest + Monte Carlo)
   162→**Estado:** NO IMPLEMENTADO
   163→**Archivos afectados:** `src/app/api/ppmt/realtime/route.ts`, `src/ppmt/risk/monte_carlo.py`
   164→
   165→Antes de permitir que el `RealtimeTrader` opere un token en modo autónomo, debe validar que el sistema tiene **edge estadísticamente significativo** sobre ese token. Actualmente el sistema puede empezar a operar sin ninguna validación previa.
   166→
   167→**Qué hacer:**
   168→- Nuevo endpoint `POST /api/ppmt/validate` que reciba `{symbol, timeframe}`
   169→- Internamente ejecuta `PortfolioBacktester.run()` con últimos 90 días
   170→- Si backtest tiene < 30 trades → verdict `INSUFFICIENT_DATA` (no operar)
   171→- Si backtest tiene ≥ 30 trades → ejecuta `MonteCarloSimulator.run()` (1000 sim)
   172→- Veredicto final: `LOW_RISK` (P95 DD <15%, RoR <5%, PF>1.3) / `MODERATE` / `HIGH_RISK`
   173→- Solo si verdict es `LOW_RISK` o `MODERATE`, se habilita el auto-trade para ese token
   174→- Guardar verdict en SQLite (`token_validations` table) con expiración de 24h
   175→
   176→#### F3. Auto-Setup ("Prepare Token") endpoint
   177→**Estado:** NO IMPLEMENTADO
   178→**Archivos afectados:** `src/app/api/ppmt/auto-setup/route.ts` (nuevo)
   179→
   180→Un token nuevo no puede operar sin: ingest → build tries → backtest → MC validation. Hoy esto requiere 4 comandos CLI manuales. Necesitamos un botón "Prepare Token" en el dashboard que ejecute toda la cadena.
   181→
   182→**Qué hacer:**
   183→- Nuevo endpoint `POST /api/ppmt/auto-setup` con body `{symbol, timeframe}`
   184→- Steps en backend: ingest 90 días → build 4 niveles trie → guardar tries → run backtest → run MC → guardar validation
   185→- Devuelve `{job_id, status: 'running'}` inmediatamente, el cliente hace polling a `/api/ppmt/auto-setup/{job_id}`
   186→- Cuando termina, actualiza el estado del token en UI a "READY_TO_TRADE"

   228→### 🟢 PRIORIDAD MEDIA — Multi-token y Multi-timeframe
   229→
   230→#### F7. Multi-token trading en vivo
   231→**Estado:** PARCIAL — `PortfolioRunner` existe pero solo se usa en backtest
   232→**Archivos afectados:** `src/ppmt/risk/portfolio_runner.py`, `src/app/api/ppmt/runner/route.ts`
   233→
   234→El `PortfolioRunner` ya orquesta múltiples `TokenEngine` pero solo en modo backtest. En modo live, solo hay un `RealtimeTrader` activo a la vez. Necesitamos múltiples traders en paralelo (uno por token), compartiendo el mismo `MoneyManager` y `ParentNodeManager`.
   235→
   236→**Qué hacer:**
   237→- Modificar `PortfolioRunner` para soportar `mode='live'` además de `'backtest'`
   238→- En modo live: un hilo por token, cada uno con su WebSocket, pero todos escribiendo al mismo `MoneyManager`
   239→- API `POST /api/ppmt/runner/start` con `{symbols: [...], mode: 'live'}`
   240→- Bloqueo de concurrencia en `MoneyManager.open_position()` para evitar race conditions
   241→
   242→#### F8. Multi-timeframe signal fusion
   243→**Estado:** NO IMPLEMENTADO
   244→**Archivos afectados:** `src/ppmt/engine/signal.py`, `src/ppmt/engine/realtime.py`
   245→
   246→Actualmente el `RealtimeTrader` opera en un solo timeframe. Un sistema profesional fusiona señales de múltiples timeframes (ej: 1h para dirección, 5m para entrada, 1m para ejecución). Sin esto, el sistema es ciego al contexto mayor.
   247→
   248→**Qué hacer:**
   249→- `SignalFusion` class que recibe señales de N timeframes y emite un score combinado
   250→- Pesos por timeframe: 1d=0.3, 4h=0.25, 1h=0.2, 15m=0.15, 5m=0.1
   251→- Solo operar si ≥ 2 timeframes alineados en dirección
   252→- Configurable por `TokenProfile` (algunos tokens operan mejor en TF alto)
   253→
   254→#### F9. ParentNodeManager UI
   255→**Estado:** PARCIAL — la clase existe en `money_manager.py` pero no hay UI
   256→**Archivos afectados:** `src/components/dashboard/parent-node-panel.tsx` (nuevo)
   257→
   258→El `ParentNodeManager` gestiona el pool de capital con nodos hijo (cada uno con `alloc_pct`, `leverage`, `auto_mode`). Hoy solo se ve en CLI. Necesitamos una UI para crear/configurar/eliminar nodos hijo y ver su rendimiento individual.
   259→
   260→**Qué hacer:**
   261→- Crear `ParentNodePanel` conectado a `/api/ppmt/parent-nodes`
   262→- Vista de árbol: Parent → Child 1, Child 2, ...
   263→- Por hijo: alloc_pct, leverage, current_value, pnl, auto_mode toggle
   264→- Botones: Add Child, Rebalance, Withdraw

   820→#### Pista #2 — MEXC v3 kline NO tiene campo `x`
   821→
   822→`_parse_mexc_kline()` hacía:
   823→```python

   825→```

   830→    "t": 1700000000000, "T": 1700003600000,
   831→    "s": "ETHUSDT", "i": "Min60",
   832→    "o": "1777.0", "c": "1778.5", "h": "1780.0", "l": "1776.0",

   834→}}, "s": "ETHUSDT", "t": 1700000000000}

   839→#### Pista #3 — MEXC requiere pings iniciados por el CLIENTE
   840→
   841→El código sólo respondía a pings del servidor:
   842→```python
   843→if msg.get("method") == "ping" or msg.get("ping"):
   844→    await ws.send(json.dumps({"method": "pong", "id": pong}))
   845→    continue
   846→```
   847→
   848→Pero MEXC v3 **no envía** pings del servidor. El protocolo es:
   849→- Cliente envía `{"method": "ping", "id": <id>}` cada ≤10s
   850→- Servidor responde `{"id": 0, "code": 0, "msg": "PONG"}` (siempre `id: 0`)
   851→

   854→#### Pista #4 — websockets library pings empeoran el problema
   855→
   856→```python
   857→async with websockets.connect(
   858→    url,
   859→    ping_interval=15,    # ← envía WebSocket control-frame pings
   860→    ping_timeout=10,     # ← espera pong control-frame en 10s
   861→    ...
   862→)

   864→

   904→#### Fix #3 — `_listen_mexc()` usa **buffered candle** strategy (`websocket_feed.py:564`)
   905→
   906→El parser por sí solo no basta: la inferencia por wall-clock es frágil cerca del límite de candle. Así que `_listen_mexc()` ahora:
   907→
   908→1. Mantiene `buffered_candle: Optional[Candle]` — el último kline del periodo actual.
   909→2. Cuando llega un kline con **timestamp distinto** al del buffer, el anterior se considera cerrado:
   910→   - Se marca `prev.closed = True`
   911→   - Se invoca `on_candle(prev)` con los valores finales (ohlcv completo)
   912→   - Se reemplaza el buffer con el nuevo kline (que abre el siguiente periodo)
   913→3. Si el timestamp es el mismo, se actualiza el buffer con los últimos valores (para tener el close/high/low/volume finales cuando llegue el cambio de timestamp).
   914→4. Al cerrar la conexión (finally), se hace flush del último buffer para no perder la última vela.
   915→
   916→Esto garantiza exactamente **una invocación de `on_candle` por periodo de vela**, con los valores finales.

   990→### Lecciones aprendidas
   991→
   992→**1. Un bug silencioso puede enmascarar otro.**
   993→El usuario llevaba horas creyendo que el motor PPMT "no generaba señales" o "estaba muy restrictivo". En realidad, **jamás llegó ninguna vela viva al motor**. El motor veía sólo 49 velas de warmup, no tenía nada con qué generar señales. Sin telemetría de "candles vivas recibidas", fue imposible diagnosticar sin auditar el código.
   994→
   995→**2. Asumir el formato de un exchange basándote en OTRO exchange es peligroso.**
   996→El parser MEXC copiaba el campo `x` de Binance. MEXC v3 no lo tiene. Acción: cada exchange debe tener tests de parser independientes que verifiquen el formato REAL de sus mensajes.
   997→
   998→**3. Las reconexiones silenciosas ocultan el problema.**
   999→El feed se reconectaba cada 30s sin log visible de "conexión cerrada". El usuario sólo veía "Connecting to MEXC" repetido, sin entender que eran reconexiones. Acción: loguear explícitamente cada close/reconnect con la razón.
  1000→
  1001→**4. Hay que distinguir protocol-level pings de application-level pings.**
  1002→WebSocket define pings a nivel de protocolo (RFC 6455 control frames). MEXC usa pings a nivel de aplicación (JSON `{"method":"ping"}`). Mezclar ambos causa desconexiones. Acción: por exchange, decidir explícitamente cuál usar y desactivar el otro.
  1003→

  1250→## v0.32.6 — State pollution entre tokens + Sweep All Tokens + UI robustez
  1251→
  1252→**Fecha:** 2026-06-17
  1253→**Commit:** próximo
  1254→**Bug reportado por el usuario:** "cuando pasas de uno a otro y vuelves a hacer un test si pasa o no para operar en el grafico se ven las operaciones que hizo en otro token y da como un error Step 0/5: error... VALIDATION RESULT asi debajo del boton"
  1255→
  1256→### Síntomas
  1257→
  1258→1. **Polución de estado entre tokens.** El usuario valida el token A (PASS o FAIL), luego selecciona el token B y vuelve a validar. En el gráfico del token B se ven los marcadores (entry/exit) del token A. Bajo el botón "Prepare Token" aparece la validación del token A ("VALIDATION RESULT" + badge PASS/FAIL del token anterior).
  1259→2. **"Step 0/5: error..." persistente.** Si una validación previa terminó en excepción, el servidor dejaba `auto_setup_status = {"step": "error", ...}` en el estado global. El frontend recibía esto por WebSocket cada segundo y lo renderizaba literalmente como `Step 0/5: error...` (porque `'error'` no está en el array `['ingesting','building','backtesting','montecarlo','done']` y `indexOf` devuelve -1, entonces `currentIdx+1 = 0`).
  1260→3. **Falta de barrido masivo.** El usuario quiere un botón que valide todos los tokens en segundo plano para descubrir cuáles son operables sin tener que hacerlo uno por uno.
  1261→4. **Historial limitado.** MEXC a veces trae menos historial del necesario; el usuario pregunta si cambiar a Binance tendría más datos.
  1262→
  1263→### Causa raíz
  1264→
  1265→**El `terminal_state` es un singleton GLOBAL compartido por todos los tokens.** Solo hay un `validation_result`, un `auto_setup_status`, un `signals_history` y un `positions` para todo el servidor. El WebSocket broadcastea este estado cada segundo a todos los dashboards conectados. Cuando el usuario cambia de token en el dashboard, el servidor NO sabe que cambió — sigue enviando el estado del token anterior.
  1266→

  1268→```js
  1269→if (s.validation_result) updateValidationResult(s.validation_result);

  1272→```
  1273→
  1274→Sin ningún filtro por token. Entonces cualquier estado global se renderizaba encima del token actual sin importar a qué token pertenecía.

  1450→### Lecciones aprendidas
  1451→
  1452→**1. Un singleton global de estado + WebSocket broadcast = polución cruzada garantizada.**
  1453→El `TerminalState` fue diseñado para un solo trader activo. Pero el dashboard permite cambiar de token sin parar el trader anterior. Esto hace que el estado del token A "contamine" la vista del token B. **Acción:** cualquier campo que pertenezca a un token específico (`validation_result`, `auto_setup_status`, `signals_history`) debe estar etiquetado con `symbol`+`timeframe` y el frontend debe filtrar por token activo antes de renderizar.
  1454→
  1455→**2. `indexOf` devuelve -1, no undefined.**
  1456→`['a','b'].indexOf('c') === -1` → `(-1)+1 = 0` → "Step 0/5: error..." en lugar de "ERROR". **Acción:** siempre verificar `currentIdx >= 0` antes de hacer aritmética con él, y manejar el caso `step === 'error'` explícitamente.
  1457→
  1458→**3. Los enums de pasos deben incluir 'error' o ser tratados como error.**
  1459→El servidor tenía 7 lugares que escribían `step: "..."` pero solo 5 valores eran esperados por el frontend. Cualquier valor fuera del enum se renderizaba como "Step 0/5". **Acción:** mantener un enum centralizado de steps válidos Y un valor `error` explícito, nunca dejar que el servidor envíe un step no reconocido.
  1460→
  1461→**4. El "Sweep All" resuelve dos problemas a la vez.**
  1462→El usuario quería descubrir qué tokens son operables sin tener que validar uno por uno. El sweep corre en background, salta los que ya tienen PASS (cached), y auto-registra como ChildNode los nuevos PASS — listo para operar. **Acción:** cualquier feature de "descubrimiento" debe ser batcheable y correr en background con polling liviano.
  1463→
  1464→### Próximos pasos recomendados
  1465→
  1466→1. **Cache de validation_result por token en el frontend.** Cuando el usuario vuelve a un token ya validado, mostrar el último resultado guardado en lugar de requerir re-validar. Necesita un endpoint `GET /api/validation/latest?symbol=X&tf=Y` que lea de la DB sin re-calcular.
  1467→2. **Auto-fallback MEXC→Binance.** Si MEXC trae <500 candles, re-intentar con Binance automáticamente. Mostrar un tag "data from Binance" en el chart.
  1468→3. **Persistir sweep results.** Guardar el resultado del último sweep en DB para que el usuario pueda cerrar y reabrir el dashboard sin perder el barrido.
  1469→4. **Sweep con timeframe múltiple.** Permitir barrer 1h Y 4h a la vez para encontrar el mejor TF por token.

  1680→### Decisiones de diseño
  1681→
  1682→**1. ¿Por qué `~/.ppmt/groups_config.json` en lugar de el proyecto?**
  1683→Los custom groups del usuario deben sobrevivir `git pull` y reinstalaciones. Si estuvieran en `ppmt/groups_config.json`, un `git pull` los sobreescribiría. Por eso:
  1684→- `ppmt/groups_config.json` = template (se versiona, solo lectura para el usuario)
  1685→- `~/.ppmt/groups_config.json` = estado real del usuario (no se versionan)
  1686→
  1687→En el primer uso, si `~/.ppmt/groups_config.json` no existe, se copia desde el template.
  1688→
  1689→**2. ¿Por qué cache 60s para tickers?**
  1690→`ccxt.fetch_tickers()` hace 1 llamada HTTP y devuelve todos los tickers del exchange (cientos). Para MEXC esto son ~600 tickers en 1 request. Cacheamos 60s para que el usuario pueda hacer click en varios grupos dinámicos sin que cada click genere una nueva llamada HTTP.
  1691→
  1692→**3. ¿Por qué `min_volume_usd=10M` para volatilidad/gainers/losers?**
  1693→Sin esto, los "Top 25 más volátiles" estarían llenos de memecoins con $1000 de volumen que hicieron +5000% en un dump. El filtro de $10M asegura que solo tokens realmente líquidos aparezcan. El usuario puede subir el filtro a $50M si quiere ser más restrictivo.
  1694→
  1695→**4. ¿Por qué no se implementó "excluir correlación > 0.8"?**
  1696→Calcular la matriz de correlación entre 50 tokens requiere:
  1697→- Cargar 50 series de precios (50 llamadas API adicionales)
  1698→- Calcular 50×50/2 = 1225 correlaciones
  1699→- Cada click en "Load group" tardaría 30+ segundos
  1700→
  1701→Para v0.33.0 se priorizó velocidad y simplicidad. Si el usuario lo pide, se puede añadir en v0.34.0 con una cache diaria.
  1702→
  1703→**5. ¿Por qué no se implementó "excluir tokens con < X días de antigüedad"?**
  1704→MEXC no expone la fecha de listing vía ccxt. Habría que llamar a `/api/v3/exchangeInfo` y parsear `onboardDate` de cada símbolo — una llamada extra. Se dejó el campo en el modelo `DEFAULT_FILTERS` (`min_listed_days`) para que el frontend lo pueda mostrar, pero el backend por ahora lo ignora (TODO v0.34.0).
  1705→
  1706→**6. ¿Por qué se mantiene el fallback a 25 majors en `/api/sweep`?**
  1707→Si el usuario hace click en "Sweep" sin seleccionar grupo (o si el endpoint `groups.resolve_group` falla por API caída), el sweep aún funciona con la lista curada. Esto es defensa en profundidad: la feature nueva nunca rompe la feature vieja.
  1708→
  1709→### Próximos pasos recomendados
  1710→
  1711→1. **Auto-fallback MEXC → Binance.** Si `resolve_group()` devuelve 0 símbolos para un grupo (porque MEXC no lista todos), re-intentar con Binance y mostrar un tag "data from Binance".
  1712→2. **Cache de validation_result por token en el frontend.** Cuando el usuario vuelve a un token ya validado, mostrar el último resultado guardado en lugar de requerir re-validar.
  1713→3. **Persistir sweep results en DB.** Tabla `sweep_results` con timestamp, group_id, symbols, verdicts — para historial.
  1714→4. **Sweep con timeframe múltiple.** Permitir barrer `1h + 5m + 15m` a la vez para encontrar el mejor TF por token.
  1715→5. **Endpoint `/api/validation/latest?symbol=X&tf=Y`.** Lee de la DB sin re-calcular, para que el dashboard muestre el último verdict instantáneamente al cambiar de token.
  1716→6. **Soporte real para `min_listed_days`.** Implementar fetch de `onboardDate` desde `exchangeInfo` y filtrar.
  1717→7. **Modo "descubrimiento".** Botón "Find New Tokens" que busca tokens listados en los últimos 7 días y los añade automáticamente a un grupo custom "recien_listados".

  1825→const sorted = results.slice().sort((a, b) => {
  1826→  const score = (r) => r.verdict === 'PASS' ? 2 : r.verdict === 'INSUFFICIENT_DATA' ? 1 : 0;
  1827→  const tier = score(b) - score(a);
  1828→  if (tier !== 0) return tier;

  1830→  const pfA = Number.isFinite(a.profit_factor) ? a.profit_factor : 0;

  1912→#### 1. Recalibración dinámica por TF — `engine/realtime.py`
  1913→
  1914→**Antes:** `recalibration_interval: int = 2000` (fijo, sin contexto TF).
  1915→
  1916→**Después:**
  1917→
  1918→```python
  1919→# Techo: en TFs altos el cálculo da valores absurdos (526 años para 1d).
  1920→# Cap a 50k velas para mantener Living Trie vivo sin recalibrar en exceso.
  1921→_RECALIBRATION_CEILING = 50_000
  1922→_RECALIBRATION_BASE = 2_000
  1923→_RECALIBRATION_REF_TF_MIN = 15  # 15m es la referencia
  1924→
  1925→def get_recalibration_interval(tf_minutes: int) -> int:
  1926→    if tf_minutes <= 0:
  1927→        return _RECALIBRATION_BASE
  1928→    factor = max(1.0, tf_minutes / _RECALIBRATION_REF_TF_MIN)
  1929→    interval = int(_RECALIBRATION_BASE * factor)
  1930→    return min(interval, _RECALIBRATION_CEILING)
  1931→```
  1932→
  1933→Tabla resultante:
  1934→
  1935→| TF  | factor | intervalo (velas) | tiempo real |
